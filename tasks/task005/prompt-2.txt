你实现的请求结果快照缓存功能模块存在一些问题，需要修复：
1. CacheKeyGenerator.generate 方法将 entities 纳入了缓存键哈希计算，导致不同实体列表的相同参数请求无法命中同一份快照，缓存复用率降低。实体列表应该只影响版本校验，而不应作为键的组成部分。
2. get_or_compute 方法存在版本竞态窗口：compute_func 执行后直接 set 存储结果，若计算期间其他线程 bump 了实体版本，旧版本结果仍被写入缓存。需要在 set 写入前重新获取最新版本号做二次校验。
3. invalidate_by_pattern 方法仅通过 _evict 清理缓存条目而不递增实体版本号，而 invalidate_by_entity 会同时清除缓存并递增版本。模式失效与实体失效的行为不一致，依赖同一实体的其他条目不会因版本校验而被动失效。需要在 _evict 中对受影响实体执行版本递增。
4. TTL 过期清理仅在 get 方法中触发，has、__contains__、len 和 get_stats 等方法均不触发过期清理，导致只写不读的过期条目持续占用内存直到 LRU 淘汰，统计信息也随之失真。需要增加独立的过期清理机制。
5. 修复后需要为该模块补充版本竞态窗口、模式失效与实体失效一致性、以及 TTL 惰性清理覆盖率的单元测试。
6. 更新 README.md 说明修复后的缓存键生成规则、版本竞态防护和过期清理策略。
7. 重新运行 poetry run pytest tests/request_snapshot_cache/ -q，需要让所有测试通过。
