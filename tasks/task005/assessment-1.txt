产物不满意：
1. cache_key.py 的 CacheKeyGenerator.generate 将 entities 纳入缓存键哈希中。key_components 同时包含 params 和 entities，导致不同实体列表键不同，get 必须传与 set 相同的实体，降低复用率。根本原因是实体被当作键身份而非版本维度。解决方案是将 entities 从 key_components 移除，仅在 check_versions_valid 使用。
2. snapshot_cache.py 的 get_or_compute 方法存在版本竞态窗口。在 compute_func 执行后直接 set 存储结果，若计算期间其他线程 bump 了版本，旧数据结果仍被写入。并发场景下可能返回脏数据，后续 get 因版本不匹配才暴露问题。根本原因是 set 写入时未重新获取最新版本号。
3. snapshot_cache.py 的 invalidate_by_pattern 仅清理缓存而未递增实体版本号。该方法遍历匹配项调用 _evict，而 _evict 只执行删除和注销操作。模式失效与实体失效行为不一致，依赖同一实体的其他条目不会因版本校验而被动失效。根本原因是 _evict 未处理实体版本生命周期。
4. snapshot_cache.py 的 TTL 过期仅在 get 中触发清理，存在隐性内存泄漏。has、__contains__、len 和 get_stats 均不触发过期清理。长期运行的系统中，只写不读的过期条目持续占用内存直到 LRU 淘汰，统计信息也随之失真。根本原因是缺乏独立过期清理机制。

过程不满意：
1. 首轮测试多个用例失败，暴露边界条件考量不足。失败涉及自定义对象序列化与测试不匹配、实体未自动初始化、重注册时旧依赖未清理、空实体列表未失效。需多轮迭代同时修改生产和测试代码才通过。降低效率并引入试错风险，根本原因是实现与测试间缺乏预设接口契约。
2. test_custom_object 修复方式是改测试而非改实现。自定义对象因 default=str 将不同实例序列化为含地址字符串，测试从跨实例一致性降级为同实例一致性。降低测试严格性，自定义对象序列化未充分验证。根本原因是迁就实现而非修正序列化策略。
3. VersionManager 与 RequestSnapshotCache 各自持有独立锁并存在嵌套调用。invalidate_by_entity 获取缓存锁后调用 VM 的 invalidate_entity，后者内部再获取 VM 锁。虽 RLock 支持同线程重入，若未来两个锁获取顺序不一致会引入死锁。根本原因是两层独立锁在调用链中交替获取。
