产物不满意：

触发节点：`checkpoint()` 方法对剩余日志条目调用 `object.__setattr__` 强制修改 LSN。实际行为：`LogEntry` 为 `frozen=True` 不可变数据类，但检查点后绕过冻结机制篡改了 LSN 字段。业务影响：破坏了追加式日志不可变的核心语义，外部引用持有的日志条目其 LSN 会无警告改变，损害审计追溯和恢复可信度。根本原因：LSN 被嵌入不可变对象内部，但又在检查点后需要重编号，导致不得不用 hack 绕过冻结。解决方案：将 LSN 从 LogEntry 中移除，改为由管理器通过列表索引位置隐式维护；或放弃 LSN 重用，改用全局单调递增编号。

触发节点：`rollback()` 和 `simulate_crash_and_recover()` 中使用 `entry.old_value is None` 判断键是否在操作前不存在。实际行为：当 SET 操作的 old_value 恰好为 None 时，回滚逻辑错误执行 delete 而非恢复为 None。业务影响：在允许 None 值作为合法数据的场景中，事务回滚和崩溃恢复会产生状态错误，将"值为 None"误判为"键不存在"导致数据丢失。根本原因：None 同时充当"键不存在"的哨兵值和合法数据值，产生语义冲突。解决方案：引入独立标记对象（如 `_MISSING = object()`）标识键不存在，与 None 值区分，并在序列化方法中处理该标记。

触发节点：`simulate_crash_and_recover()` 第 182 行定义的局部变量 `start_lsn`。实际行为：该变量被计算赋值后从未被引用，方法直接遍历全部日志条目，未使用该变量做范围过滤。业务影响：死代码增加维护困惑，后续开发者可能误以为可依赖该变量做筛选，引入隐藏缺陷。根本原因：开发中计划用 `start_lsn` 过滤日志范围，但发现检查点后日志已截断无需筛选，却忘记删除变量。解决方案：删除未使用的 `start_lsn` 变量，在方法注释中明确说明检查点后日志已截断故直接遍历全量即可。

触发节点：`checkpoint()` 方法第 169 行和第 172 行存在相同赋值 `self._next_lsn = len(self._log)`。实际行为：日志截断后和 LSN 重编号后各执行一次相同赋值，中间操作不改变条目数量故结果一致。业务影响：冗余代码降低可读性，让审查者质疑是否有逻辑遗漏，增加维护心智负担。根本原因：先在第 169 行设置 `_next_lsn`，后增加重编号循环时出于谨慎再次赋值但未删除前一句。解决方案：删除第 169 行的 `_next_lsn` 赋值，仅保留 LSN 重编号后的最终赋值即可。

过程不满意：

触发节点：`TestCrashRecovery` 多个测试用 `_` 忽略 `simulate_crash_and_recover()` 的 `redo_count` 和 `undo_count` 返回值。实际行为：测试仅断言恢复后的状态键值内容，部分验证了 `undo >= 1`，但未对所有场景的 REDO/UNDO 计数做精确断言。业务影响：计数是恢复过程的元信息，若计数错误说明 REDO 或 UNDO 阶段有重复或遗漏操作，当前测试无法探测此类缺陷。根本原因：测试设计只关注"最终状态是否正确"而忽略"恢复过程是否精确"。解决方案：为每个恢复测试补充精确的计数断言，例如断言某场景 redo 和 undo 的具体数值，在综合场景中也验证计数正确性。

触发节点：整个测试套件缺少将 `None` 作为合法存储值进行 SET、回滚和恢复的边界测试。实际行为：所有测试从未将 `None` 作为存储值传入，也未测试 `old_value` 为 None 但键原本存在且值恰好为 None 的回滚场景。业务影响：上述哨兵值冲突缺陷因缺乏测试覆盖而完全不可见，当应用层实际使用 None 值时会暴露回滚和恢复错误。根本原因：测试默认假设存储值为非 None 的普通数据，未将 None 视为边界条件，也未意识到代码中 None 的语义双重性。解决方案：新增用例覆盖：事务 SET None 后回滚、无事务 SET None 后 DELETE、崩溃恢复中含 None 值的已提交及未提交事务。
