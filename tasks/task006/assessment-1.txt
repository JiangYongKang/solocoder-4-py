产物不满意：
1. log_entry.py 的 checkpoint 方法对剩余日志条目调用 object.__setattr__ 强制修改 LSN。LogEntry 为 frozen=True 不可变数据类，但检查点后绕过冻结机制篡改了 LSN 字段。破坏了追加式日志不可变的核心语义，外部引用持有的日志条目其 LSN 会无警告改变，损害审计追溯和恢复可信度。根本原因是 LSN 被嵌入不可变对象内部，但又在检查点后需要重编号，导致不得不用 hack 绕过冻结。应将 LSN 从 LogEntry 中移除改为由管理器通过列表索引位置隐式维护，或放弃 LSN 重用改用全局单调递增编号。
2. state_store.py 的 rollback 和 simulate_crash_and_recover 方法中使用 entry.old_value is None 判断键是否在操作前不存在。当 SET 操作的 old_value 恰好为 None 时，回滚逻辑错误执行 delete 而非恢复为 None。在允许 None 值作为合法数据的场景中，事务回滚和崩溃恢复会产生状态错误，将值为 None 误判为键不存在导致数据丢失。根本原因是 None 同时充当键不存在的哨兵值和合法数据值产生语义冲突，应引入独立标记对象标识键不存在。
3. state_store.py 的 simulate_crash_and_recover 方法定义了一个局部变量 start_lsn 被计算赋值后从未被引用，方法直接遍历全部日志条目，未使用该变量做范围过滤。死代码增加维护困惑，后续开发者可能误以为可依赖该变量做筛选。根因是开发中计划用 start_lsn 过滤日志范围，但发现检查点后日志已截断无需筛选，却忘记删除变量。
4. state_store.py 的 checkpoint 方法中存在相同赋值的冗余代码：日志截断后和 LSN 重编号后各执行一次相同的 _next_lsn 赋值，中间操作不改变条目数量故结果一致。冗余代码降低可读性，让审查者质疑是否有逻辑遗漏。根本原因是在重编号循环时出于谨慎再次赋值但未删除前一句。

过程不满意：
1. TestCrashRecovery 多个测试用下划线忽略 simulate_crash_and_recover 的返回值。测试仅断言恢复后的状态键值内容，部分验证了 undo 计数，但未对所有场景的 REDO/UNDO 计数做精确断言。计数是恢复过程的元信息，若计数错误说明 REDO 或 UNDO 阶段有重复或遗漏操作，当前测试无法探测此类缺陷。根因是测试设计只关注最终状态是否正确而忽略恢复过程是否精确。
2. 整个测试套件缺少将 None 作为合法存储值进行 SET、回滚和恢复的边界测试。所有测试从未将 None 作为存储值传入，也未测试 old_value 为 None 但键原本存在且值恰好为 None 的回滚场景。上述哨兵值冲突缺陷因缺乏测试覆盖而完全不可见。根因是测试默认假设存储值为非 None 的普通数据，未将 None 视为边界条件。
